Claude Pipeline Guide

A repeatable, agent-driven delivery pipeline for software features — from design to deployed code — using three slash commands.

The Pipeline

Every feature or fix moves through three stages. Each command hands off structured artifacts to the next — no implicit context, no re-explaining.

/plan Design
→
/cook Implement
→
/fix Repair

How They Fit Together

  • /plan — Takes a feature description and produces a structured plan directory (plans/YYMMDD-{slug}/) with a plan.md and numbered phase files.
  • /cook — Reads an existing plan directory and implements each phase sequentially, spawning specialist sub-agents per phase.
  • /fix — Takes a broken test or error message and scouts, diagnoses, patches, and verifies a fix without touching unrelated code.

Design principle: Commands are stateless. They read files, write files, and exit. The filesystem is the only shared state between stages.

Repository Layout

.claude/
  agents/     # sub-agents spawned by commands
  commands/   # slash commands (/plan, /cook, /fix)
  skills/     # behavioral guidance loaded by skill system
  rules/      # path-scoped design principles
  hooks/      # shell hooks for lifecycle events
plans/        # generated by /plan — one folder per feature
◇ Command 01

/plan

Turns a natural-language feature description into a structured, versioned plan directory that /cook can execute without additional context.

Usage

/plan [--fast | --hard | --parallel | --two] <description>

Output Structure

plans/YYMMDD-{slug}/
  plan.md              # overview, phase checklist, risks
  phase-01-{name}.md   # per-phase implementation detail
  phase-02-{name}.md
  ...

Modes

--fast --hard --parallel --two
  • --fast — Immediate plan, no research agents. For small well-understood tasks.
  • --hard — Spawns researcher sub-agents before planning. For architectural decisions or unfamiliar territory.
  • --parallel — Each phase file gets an exclusive file ownership list for simultaneous implementation without conflicts.
  • --two — Produces two competing plans from different research agents. Human picks one; planner finalizes plan.md.

Agent Chain (Hard mode)

R
plan-researcher
Investigates approach, reads codebase, returns structured findings
P
planner
Synthesizes research into plan.md + numbered phase files
V
plan-reviewer
Red-teams phase files for gaps, vague steps, security holes

plan.md Structure

# Plan: {Feature Name}
Status: In Progress
Date: YYYY-MM-DD
Mode: Hard

## Overview       — what this plan delivers and why
## Phases         — checkbox list linking to phase files
## Research       — researcher findings and chosen approach
## Dependencies   — prerequisite work or external services
## Risks          — HIGH/MEDIUM/LOW with mitigations

Rule: Phase files must be detailed enough that implementation can begin without asking. Vague steps like "implement the service" are rejected by the plan-reviewer agent.

◇ Command 02

/cook

Reads an existing plan directory and executes it phase by phase, spawning specialist sub-agents and tracking completion in the plan checklist.

Usage

/cook plans/260418-my-feature/plan.md

Execution Model

Each unchecked phase in plan.md executes in order. After a phase completes successfully, /cook marks it [x] before proceeding. If a phase fails, execution halts — no partial phases are silently skipped.

Phase Lifecycle

1 Read phase file, extract requirements and implementation steps
2 Implement code following codebase conventions
3 Run success criteria — tester agent validates
4 Mark phase [x] in plan.md, proceed to next phase

Sub-agents Spawned

T
tester
Runs success criteria, reports pass/fail per criterion
D
debugger
Called on failure — diagnoses root cause and patches before re-testing
G
git-manager
Stages files, creates conventional commit after each completed phase

Modes

default --auto --fast --parallel
  • default — Interactive: pauses at each Review Gate for human approval.
  • --auto — Auto-approve if code review score ≥ 9.5 with 0 critical issues.
  • --fast — Single-pass implement, no tester/reviewer loop.
  • --parallel — Multi-agent execution for 3+ independent phases.

Important: /cook never writes to a phase file after completion. The plan directory is ground truth — do not edit phase files while a cook run is in progress.

◇ Command 03

/fix

Takes a failing test, error message, or bug description and runs a structured scout → debugger → review pipeline to produce a minimal, verified patch.

Usage

/fix "AuthService_Login_Returns401_WhenPasswordInvalid"
/fix "TypeError: Cannot read properties of undefined (reading 'id')"
/fix --quick    # skip review step
/fix --auto     # non-interactive, apply first valid fix
/fix --review   # force human review before applying patch

Modes

default --auto --quick --review
  • default — Full pipeline with human confirmation before patch is applied.
  • --auto — Non-interactive. Applies first fix that passes tests. Suited for CI.
  • --quick — Skips review agent. Scout + debugger + apply only. Faster, less safe.
  • --review — Forces review even when the main agent would skip it. Use on high-risk paths.

Agent Pipeline

S
scout
Locates the failing test, reads related source files, maps call graph
D
debugger
Identifies root cause, writes minimal diff, re-runs test to confirm green
R
code-reviewer
Checks patch doesn't break adjacent tests or introduce regressions

Constraints

  • Patch touches only files identified by the scout agent — no scope creep.
  • If the fix requires changes to more than 3 files, /fix warns and asks for confirmation.
  • No refactoring during a fix run. If refactoring is needed, it is noted and deferred to a /plan → /cook cycle.

When to use /fix vs /cook: Use /fix for regressions and bugs. Use /cook for new features. If you find yourself using /fix to add behavior, stop — create a plan instead.
